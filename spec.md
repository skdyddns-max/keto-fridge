# 키토 냉장고 (Keto Fridge) — 프로젝트 스펙 v1.0

> 집 냉장고에 있는 재료만으로 키토 식단을 지속할 수 있게 돕는 웹앱.
> 실행 중 AI 호출 없음 → 완전 무료 배포 · API 비용 0원.

---

## 0. 핵심 원칙 (이 프로젝트의 헌법)

1. **런타임에 외부 API를 호출하지 않는다.** 모든 매칭·계산은 클라이언트(브라우저) 로컬에서 수행한다.
2. **영양값은 저장하지 않고 계산한다.** 레시피는 재료+그램수만 갖고, 순탄수/매크로는 재료 영양표에서 빌드 시점에 자동 산출한다.
3. **가독성이 최우선이다.** 정보가 많아도 사용자가 한눈에 "지금 뭘 해먹을 수 있는지" 파악되어야 한다.
4. 파일명은 영어만 사용한다.
5. 언어는 한국어 단일.

---

## 1. 제품 정의

- **타깃:** 키토를 지속하려는 일반 사용자(공개 배포). 1인 가구 포함.
- **해결하는 문제 (우선순위):**
  1. (1순위) 냉장고 재료로 **뭘 해먹을지** — 재료→레시피 매칭
  2. (2순위) 매번 비슷한 것만 먹어 **질리는** 문제 — 다양성(레시피 800개 + 다양성 랭킹)
  3. (3순위) 이게 **키토에 맞나** — 순탄수 판정
- **핵심 흐름:**
  재료 입력 → 키토 레시피 추천(지금 가능/거의 가능) → 순탄수·매크로 표시 → 레시피 상세(조리 순서)

---

## 2. 키토 도메인 규칙

- **키토 기준:** 하루 총 순탄수 **20g 이하**.
- **순탄수 = 총탄수 − 식이섬유** (재료 영양표에서 계산).
- **레시피 단위 처리:** MVP엔 하루 누적 트래킹이 없으므로, 하루 2~3끼를 합쳐도 20g 안에 들도록 **1인분당 순탄수 ≤ 8g**인 레시피를 "키토 적합"으로 채택한다. (8g × 2끼 = 16g, 여유 확보)
- **매크로 목표:** 지방 70% / 단백질 25% / 탄수 5% (칼로리 기준). 레시피별로 실제 매크로를 파이차트로 표시.
- **제외 재료 기능:** 사용자가 특정 재료(예: 유제품)를 제외 목록에 넣으면, 해당 재료가 든 레시피는 추천에서 제거한다.

## 3. 데이터 설계

### 3-1. 재료 영양표 `ingredients.json`
레시피에 등장하는 재료(약 150~250종)만 수록. 식약처/USDA 기준.

```json
{
  "id": "pork_belly",
  "name": "삼겹살",
  "aliases": ["삼겹", "돼지삼겹살"],
  "per100g": { "kcal": 331, "carbG": 0, "fiberG": 0, "fatG": 28, "proteinG": 17 },
  "keto": "friendly"          // friendly | caution | avoid
}
```

### 3-2. 레시피 `recipes.json` (목표 800개)
`computed` 필드는 손으로 적지 않는다 — 빌드 스크립트가 채운다.

```json
{
  "id": "r0001",
  "name": "삼겹살 양배추 버터볶음",
  "category": "meat",         // meat | egg | seafood | veg_side | soup | salad | snack ...
  "servings": 1,
  "ingredients": [
    { "id": "pork_belly", "name": "삼겹살", "grams": 150 },
    { "id": "cabbage",    "name": "양배추", "grams": 100 },
    { "id": "butter",     "name": "버터",   "grams": 10 }
  ],
  "steps": ["팬에 버터를 녹인다", "삼겹살을 굽는다", "양배추를 넣고 볶는다"],
  "tags": ["돼지고기", "볶음", "10분"],
  "computed": { "kcal": 0, "netCarbG": 0, "fatG": 0, "proteinG": 0 }
}
```

### 3-3. 빌드 검증 스크립트 `scripts/build-data.ts`
1. 모든 레시피 재료가 `ingredients.json`에 존재하는지 확인 (없으면 에러).
2. `computed` 자동 계산: `netCarbG = Σ (carbG − fiberG) × grams / 100`, 매크로/kcal 동일 방식.
3. 키토 필터: `netCarbG > 8`인 레시피는 별도 로그로 표시하고 `keto:false` 태깅(추천에서 자동 제외).
4. 중복 레시피(이름·재료 조합) 감지.
5. 카테고리 분포 리포트 출력(다양성 확인용).

## 4. 매칭 엔진 (앱의 핵심 IP) `src/lib/match.ts`

입력: 사용자가 가진 재료 집합 `owned`, 제외 재료 `excluded`.

각 레시피에 대해:
- `required` = 레시피 재료 집합 (조미료류는 "기본 보유"로 간주 옵션)
- `coverage = |owned ∩ required| / |required|`
- **제외 필터:** `required ∩ excluded ≠ ∅` 이면 탈락
- **키토 필터:** `keto:false` 탈락

결과 상태 3단계:
| 상태 | 조건 | 표시 |
|------|------|------|
| `cookNow` | coverage = 100% | "지금 만들 수 있어요" |
| `almost` | 부족 재료 1~2개 | "이것만 추가하면 돼요: 양파, 버터" |
| `explore` | 그 외 | 접힌 영역/더보기 |

**랭킹:** `cookNow` 우선 → 부족 개수 적은 순 → **다양성 보정**(같은 category가 연속 노출되지 않도록 섞어서, 상위 결과의 category 다양성 확보). 800개 풀 + 다양성 랭킹으로 "매번 똑같다" 느낌을 방지.

## 5. 기능 범위

### MVP (Phase 1~3, 첫 배포)
- [필수] 재료 입력 (자동완성 태그 입력, aliases 인식)
- [필수] 레시피 추천 (cookNow / almost 구분)
- [필수] 순탄수 + 매크로 파이차트 표시
- 레시피 상세 (조리 순서)
- "이 재료만 추가하면 됨" 안내 (almost 상태)
- 제외 재료 설정 (localStorage 저장)
- 즐겨찾기 (localStorage 저장, 로그인 불필요)

### Later (Phase 4, 배포 후)
- Supabase 계정 · 기기간 즐겨찾기 동기화
- 하루 순탄수 누적 트래킹
- 장보기 리스트

## 6. 기술 스택

- **프론트:** React + Vite + TypeScript + Tailwind CSS
- **상태:** 로컬 상태 + `localStorage` (즐겨찾기·제외재료·최근 입력)
- **데이터:** 정적 JSON 번들 (`ingredients.json`, `recipes.json`). 런타임 백엔드 없음.
- **차트:** 매크로 파이 — 경량 SVG 직접 구현 또는 recharts.
- **배포:** Vercel(Hobby, 무료) 또는 Cloudflare Pages(무료). 정적 빌드만 업로드.
- **PWA:** manifest + service worker로 홈화면 추가 · 오프라인 지원(정적 데이터 캐싱).
- **Supabase:** 스택에 예약. **Phase 4 전까지 연결하지 않음.** (무료 유지)

---

## 7. 디자인 방향 (frontend-design 스킬 사용)

17번을 위임하셨으므로 아래를 출발점으로 하되 Claude Code가 다듬는다.
- **톤:** 클린·미니멀, 음식/재료 중심. 정보 밀도 높아도 여백으로 숨 쉬게.
- **가독성 최우선:** 큰 탭 타깃(모바일 PWA), 명확한 위계(지금 가능 > 거의 가능), 순탄수는 굵고 크게.
- **타이포:** Pretendard 우선, 보조 Noto Sans KR.
- **매크로 차트:** 작지만 명료하게. 색으로 지방/단백질/탄수 즉시 구분.

## 8. 단계별 계획 (Phase)

- **Phase 0 — 데이터:** `ingredients.json` 구축 → 레시피 생성 파이프라인 → 800개 배치 생성 → `build-data.ts` 검증. (가장 큰 작업)
- **Phase 1 — 뼈대:** 프로젝트 스캐폴드, 재료 입력 UI, 매칭 엔진, 결과 리스트(순탄수·매크로).
- **Phase 2 — 완성도:** 레시피 상세, almost 안내, 제외재료 설정, 즐겨찾기, 다양성 랭킹, 반응형·가독성.
- **Phase 3 — PWA·배포:** manifest/SW, 오프라인 캐싱, Vercel 무료 배포.
- **Phase 4 — Later:** Supabase 계정, 동기화, 하루 트래킹.

---

## 9. 리스크

1. **레시피 800개 품질/정확도** — 최대 난관. 순탄수를 재료에서 계산하므로 숫자 신뢰도는 확보되나, 레시피 자체의 다양성·현실성은 배치 생성 + 검토 필요.
2. **재료 영양표 정확도** — 키토 판정의 근간. 식약처/USDA 기준으로 검증, "참고용" 고지 문구 필요.
3. **범위 팽창** — 5개 기능을 한 번에 넣되 가독성이 무너지지 않게. Phase 순서를 지켜 하나씩 안정화.
4. **자동완성 재료 인식** — aliases/오타 처리 미흡 시 매칭이 안 됨. 입력 정규화 로직 필요.
